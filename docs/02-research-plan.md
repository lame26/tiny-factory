# 조사 계획

## 조사 목표

이번 조사의 목적은 초기 컨셉을 확정하는 것이 아니라, `01-initial-game-concept.md`의 가설을 검증하고 보강하는 것이다. 조사는 세 가지 질문에 답해야 한다.

1. Eatventure의 어떤 작동 원리가 반복 플레이와 성장감을 만드는가?
2. 그 원리를 전자/가전 생산 게임으로 바꿀 때 무엇을 유지하고 무엇을 바꿔야 하는가?
3. Steam PC용 제품으로 만들기 위해 모바일 레퍼런스에서 어떤 부분을 확장하거나 제거해야 하는가?

## 조사 우선순위

1순위: Eatventure의 구조적 분석

- 코어 루프
- 업그레이드 주기
- 자동화 해금
- 스테이지 진행
- 직원/장비/경제의 관계
- 세션 길이와 반복 동기

2순위: Steam 유사 장르 분석

- 운영 게임
- 자동화 게임
- 캐주얼 타이쿤
- 방치형 또는 반자동 성장 게임
- 공장/생산 테마 게임

3순위: 전자/가전 테마 적합성 분석

- 제품군 후보
- 공정 단계 후보
- 시각적으로 구분 가능한 작업
- 플레이어가 즉시 이해할 수 있는 생산 흐름

4순위: 시장/리뷰 분석

- Steam 리뷰에서 반복적으로 언급되는 장점/불만
- 모바일 레퍼런스가 PC로 왔을 때의 기대 차이
- 가격, 콘텐츠 볼륨, 업데이트 기대

## 조사 항목 목록

### Eatventure 분석 항목

- 첫 10분 플레이 흐름
- 첫 자동화 도달 시점
- 업그레이드 항목 종류
- 직원의 역할과 체감 가치
- 수익 상승 곡선의 체감 방식
- 매장 또는 지역 전환 방식
- 이벤트, 장비, 메타 시스템의 역할
- 광고/과금 구조가 게임 루프에 끼치는 영향
- UI가 플레이어에게 다음 행동을 알려주는 방식

### 유사 게임 분석 항목

- Steam에서 긍정 평가를 받는 운영/자동화 게임의 공통점
- 부정 리뷰에서 반복되는 피로 요소
- 너무 단순하다는 평가와 너무 복잡하다는 평가의 기준
- 자동화가 재미로 느껴지는 시점
- 반복 플레이의 목표 구조
- 장기 성장 시스템의 형태

### 전자/가전 테마 분석 항목

- 플레이어가 익숙하게 알아볼 수 있는 제품군
- 제품군별 최소 공정 단계
- 부품, 조립, 검사, 포장, 출고의 시각적 표현 가능성
- 제품 단가 상승이 자연스러운 순서
- 공장 확장과 제품 라인 확장이 연결되는 방식
- 테마가 제공하는 고유 시스템 후보: 결함률, 품질 등급, 모듈 조합, 리콜, 신제품 출시, 자동 검사 등

## 조사 대상 소스 예시

### 공식 스토어

- App Store, Google Play의 Eatventure 소개, 스크린샷, 업데이트 노트
- Steam의 유사 운영/자동화 게임 스토어 페이지
- 게임별 태그, 유저 리뷰, 스크린샷, 트레일러 설명

활용 기준:

- 공식 설명은 게임이 스스로 강조하는 핵심 판매 포인트를 파악하는 데 사용한다.
- 스크린샷과 트레일러는 UX 밀도, 화면 구성, 진행 판타지를 관찰하는 데 사용한다.
- 마케팅 문구를 그대로 기획 근거로 쓰지 않고, 실제 시스템 확인 자료와 대조한다.

### 위키

- Eatventure 관련 위키
- 유사 게임의 시스템 위키
- 아이템, 장비, 업그레이드, 이벤트, 지역 목록

활용 기준:

- 시스템 목록화와 구조 파악에 사용한다.
- 정확한 수치가 최신인지 검증이 필요하므로, 단일 위키만으로 밸런스 근거를 확정하지 않는다.

### 커뮤니티

- Reddit
- Steam 커뮤니티
- Discord 공개 채널이 있는 경우
- 게임별 포럼

활용 기준:

- 반복 플레이 동기, 불만, 막히는 지점, 인기 전략을 파악한다.
- 특정 유저 의견을 절대 기준으로 삼지 않고, 반복 패턴만 추출한다.

### 리뷰

- Steam 유저 리뷰
- 모바일 스토어 리뷰
- YouTube 리뷰 또는 장르 소개 영상
- 블로그/웹진 리뷰

활용 기준:

- "재미있다/지루하다"보다 그 이유를 추출한다.
- 플레이 시간, 가격, 반복성, 업데이트, 조작감 관련 언급을 우선 본다.

### 공략

- 초보자 가이드
- 최적 업그레이드 순서
- 이벤트 공략
- 장비 티어 리스트

활용 기준:

- 실제 플레이어가 중요하게 여기는 의사결정이 무엇인지 파악한다.
- 최적화가 단일 정답으로 굳어지는지, 다양한 전략이 가능한지 확인한다.

### 영상 설명

- 첫 플레이 영상
- 장기 플레이 영상
- 업데이트 소개 영상
- 이벤트 플레이 영상

활용 기준:

- 텍스트로 드러나지 않는 조작 리듬, 화면 가독성, 성장 체감을 관찰한다.
- 첫 5분, 첫 30분, 중후반 화면을 분리해 기록한다.

### 인터뷰/개발자 코멘트

있다면 다음을 확인한다.

- 개발자가 의도한 핵심 루프
- 업데이트 방향
- 수익화 또는 플랫폼 전략
- 유저 피드백을 반영한 변경점

활용 기준:

- 의도와 실제 유저 반응의 차이를 보는 보조 자료로 사용한다.

## 조사 시 확인해야 할 질문

### 레퍼런스 구조 질문

- 플레이어가 처음 이해해야 하는 행동은 몇 개인가?
- 첫 업그레이드까지 걸리는 시간은 어느 정도인가?
- 자동화가 해금되기 전과 후의 재미가 어떻게 달라지는가?
- 새 스테이지가 열릴 때 실제로 달라지는 것은 무엇인가?
- 수익 배율 상승이 단조롭게 느껴지지 않게 하는 장치는 무엇인가?

### PC 전환 질문

- 모바일 화면과 PC 화면의 정보 밀도 차이를 어떻게 처리할 수 있는가?
- 마우스 조작에서 직접 이동 조작이 재미있을지, 불편할지 확인해야 한다.
- Steam 유저가 반복 플레이를 납득하려면 어떤 추가 깊이가 필요한가?
- 광고 보상에 의존한 속도감을 PC에서는 어떤 시스템으로 대체할 수 있는가?

### 테마 치환 질문

- 음식의 "즉시 소비"를 전자제품의 "구매/출고/납품"으로 바꿔도 보상감이 유지되는가?
- 조리 과정을 조립 과정으로 바꾸면 공정이 너무 복잡해지지 않는가?
- 제품군별 차이가 시각적으로 읽히는가?
- 결함, 검사, 품질 같은 요소가 재미가 되는가, 피로가 되는가?

## 참고 정보와 직접 치환 근거의 구분

참고만 할 정보:

- 원작의 구체 수치
- 특정 UI 배치
- 특정 장비 이름
- 특정 이벤트 명칭
- 특정 캐릭터 외형
- 특정 과금 패키지 구성

직접 치환의 근거로 쓸 수 있는 정보:

- 플레이어가 이해하는 행동 순서
- 생산-판매-재투자 루프의 길이
- 자동화 해금이 주는 역할 변화
- 업그레이드 카테고리 간 관계
- 반복 플레이에서 지루함을 줄이는 진행 단위
- 유저 리뷰에서 반복적으로 나타나는 기대와 불만
