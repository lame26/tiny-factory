# 레퍼런스 추출 프레임워크

## 목적

이 문서는 Eatventure를 그대로 베끼기 위한 문서가 아니다. 레퍼런스의 표면 요소와 구조적 원리를 분리하고, 우리 게임에 적용 가능한 원리만 추출하기 위한 분석 지침서다.

분석 목표는 다음과 같다.

- 무엇을 모사하면 IP/차별화 위험이 생기는지 구분한다.
- 어떤 원리는 장르 문법으로 참고할 수 있는지 정리한다.
- 전자/가전 생산 게임으로 바꿀 때 필요한 변형 기준을 만든다.

## 분석 층위

### 1. 표면층

표면층은 플레이어가 즉시 보는 소재와 표현이다.

조사할 것:

- 음식, 매장, 캐릭터, 의상, 배경, 색상, 아이콘
- 제품명, 스테이지명, 이벤트명
- 화면 구성의 인상
- 애니메이션 리듬과 캐릭터 비율

그대로 모사하면 위험한 요소:

- 음식점 테마를 단어만 바꿔 같은 흐름으로 재현
- 유사한 캐릭터 실루엣, UI 배치, 아이콘 스타일
- 같은 지역/매장 진행 감각
- 원작 고유의 장비, 이벤트, 보상 명칭

원리만 차용해도 되는 요소:

- 친숙한 소재를 사용해 첫 이해를 빠르게 만드는 방식
- 생산 결과물이 시각적으로 분명해야 한다는 원칙
- 새 영역이 열릴 때 화면 변화가 즉시 보여야 한다는 원칙

우리 게임 반영 기준:

- 음식과 매장은 전자제품 생산 라인, 조립대, 검사대, 출고대, 쇼룸으로 바꾼다.
- 캐릭터/장비/공간의 시각 문법은 공장, 작업장, 가전 브랜드, 물류센터 쪽에서 새로 설계한다.

### 2. 시스템층

시스템층은 플레이어 행동과 게임 규칙의 관계다.

조사할 것:

- 플레이어 직접 이동과 작업 처리 방식
- 생산 지점 개방 순서
- 직원 고용과 자동화 조건
- 업그레이드 종류
- 새 스테이지 진입 조건
- 직접 조작과 자동 처리의 비율 변화

그대로 모사하면 위험한 요소:

- 동일한 작업 순서, 동일한 해금 타이밍, 동일한 업그레이드 묶음
- 원작의 스테이지 구조를 전자제품 이름으로만 바꾸는 것

원리만 차용해도 되는 요소:

- 플레이어가 처음에는 직접 처리하고, 이후 직원/자동화가 대체하는 역할 변화
- 생산 능력, 이동 능력, 판매 가치가 서로 맞물리는 구조
- 새 생산 지점이 열릴수록 동선과 우선순위가 바뀌는 구조

우리 게임 반영 기준:

- 전자/가전 테마의 공정성을 살려 부품 공급, 조립, 검사, 포장, 출고 중 어떤 단계를 플레이어 행동으로 둘지 별도 설계한다.
- 자동화는 단순 직원 증가뿐 아니라 컨베이어, 로봇 팔, 자동 검사기, 분류기 같은 설비 중심으로 확장한다.

### 3. 경제층

경제층은 수익, 비용, 업그레이드, 진행 속도의 관계다.

조사할 것:

- 수익 발생 주기
- 업그레이드 가격 상승 방식
- 생산 속도와 판매 가격의 체감
- 병목이 생기는 지점
- 이벤트 또는 메타 보너스가 경제에 미치는 영향
- 광고/과금 보상이 속도에 끼치는 영향

그대로 모사하면 위험한 요소:

- 원작의 구체 수치, 배율, 가격 곡선을 재사용
- 동일한 업그레이드 비용 구조와 보상 타이밍
- 모바일 과금 구조를 PC에 무비판적으로 이식

원리만 차용해도 되는 요소:

- 작은 보상을 자주 주고, 큰 목표를 중간중간 제시하는 리듬
- 업그레이드 직후 체감 차이가 분명해야 한다는 원칙
- 특정 병목을 해소하면 전체 생산성이 올라가는 구조

우리 게임 반영 기준:

- 경제 수치는 새로 설계한다.
- PC 버전에서는 광고 보상 대신 연구, 계약, 생산 목표, 효율 보너스 등 플레이 기반 보상으로 속도감을 만든다.
- 업그레이드는 가격, 속도, 품질, 자동화, 운반량, 결함률 감소처럼 테마와 연결한다.

### 4. UX층

UX층은 플레이어가 다음 행동을 이해하고 조작하는 방식이다.

조사할 것:

- 화면에서 다음 목표를 알려주는 방식
- 업그레이드 버튼의 위치와 빈도
- 수익, 비용, 진행 조건의 표시 방식
- 조작 피로도
- 자동화 이후 관찰 화면의 가독성
- 튜토리얼 개입 정도

그대로 모사하면 위험한 요소:

- UI 레이아웃, 버튼 위치, 아이콘, 색상 체계를 실질적으로 복제
- 같은 알림 방식과 진행 표시 방식을 그대로 사용하는 것

원리만 차용해도 되는 요소:

- 다음에 살 수 있는 업그레이드를 명확히 보여주는 원칙
- 현재 병목과 목표를 플레이어가 빠르게 이해해야 한다는 원칙
- 수익 증가가 숫자와 화면 변화로 동시에 체감되어야 한다는 원칙

우리 게임 반영 기준:

- PC에 맞는 더 넓은 화면, 패널, 툴팁, 단축 조작을 고려한다.
- 생산 라인 상태, 주문 대기, 병목, 직원 배치가 한눈에 보이도록 별도 UX를 설계한다.

### 5. 메타층

메타층은 한 세션을 넘어 플레이를 지속하게 만드는 구조다.

조사할 것:

- 다음 지역 또는 매장으로 넘어가는 방식
- 장비, 카드, 캐릭터, 영구 보너스
- 이벤트와 시즌 구조
- 반복 플레이 목표
- 장기 성장의 리셋 또는 누적 방식

그대로 모사하면 위험한 요소:

- 동일한 장비 슬롯, 등급, 이벤트 구조, 보상 체계
- 지역 이동 구조를 동일하게 가져오는 것

원리만 차용해도 되는 요소:

- 세션 내 성장과 세션 간 성장이 분리되어야 한다는 원칙
- 다음 목표가 항상 보여야 한다는 원칙
- 반복 플레이의 이유가 단순 수치 증가만은 아니어야 한다는 원칙

우리 게임 반영 기준:

- 연구 개발, 브랜드 성장, 제품 라인 특화, 공장 이전, 계약 등 전자/가전 테마에 맞는 메타를 설계한다.
- 이벤트는 "한정 메뉴"가 아니라 "박람회 주문", "신제품 런칭", "긴급 납품", "리콜 대응" 같은 형태가 자연스럽다.

### 6. 상품층

상품층은 플랫폼, 가격, BM, 업데이트, 패키징에 관한 영역이다.

조사할 것:

- Eatventure의 모바일 BM 구조
- 유사 Steam 게임의 가격대와 콘텐츠 볼륨
- DLC, 업데이트, 얼리 액세스 활용 여부
- 리뷰에서 가격 대비 만족도가 언급되는 방식

그대로 모사하면 위험한 요소:

- 광고 보상, 모바일식 과금 압박, 랜덤 상품 구조를 Steam에 그대로 이식
- 모바일 세션 설계를 PC 패키지 상품으로 그대로 포장

원리만 차용해도 되는 요소:

- 짧은 목표와 장기 목표를 병렬로 제시하는 상품 구조
- 업데이트로 이벤트, 제품군, 공장을 확장하기 쉬운 구조

우리 게임 반영 기준:

- Steam용으로는 기본 구매형, 얼리 액세스, 데모, 업데이트 로드맵을 별도로 검토한다.
- BM은 게임 루프 설계 이후 결정하며, 초기 기획 단계에서 과금 중심으로 루프를 설계하지 않는다.

## 분석 결과 반영 기준

분석 결과는 다음 기준을 통과해야 우리 게임에 반영할 수 있다.

- 원작의 고유 표현이 아니라 장르적 원리인가?
- 전자/가전 테마로 바꿨을 때 더 자연스러워지는가?
- Steam PC 환경에서 조작과 정보 표시가 개선될 여지가 있는가?
- 단순 반복이 아니라 장기 성장 또는 자동화 의사결정으로 확장되는가?
- IP 위험을 피할 만큼 명확한 표현, 구조, 명칭, 진행 방식의 차이가 있는가?

## 레퍼런스 추출 체크리스트

각 관찰 항목마다 다음 질문에 답한다.

- 이것은 표면 표현인가, 구조적 원리인가?
- 원작 고유의 명칭, 이미지, UI, 수치에 의존하는가?
- 같은 효과를 전자/가전 테마에서 다르게 표현할 수 있는가?
- 우리 게임의 핵심 판타지인 생산, 조립, 판매, 자동화와 연결되는가?
- PC 조작과 화면에 맞게 확장할 수 있는가?
- 그대로 쓰면 유사성이 강하게 느껴지는가?
- 변형한다면 어떤 시스템으로 바꿀 수 있는가?
- 이 요소가 초반 이해, 중반 성장, 후반 반복 중 어디에 기여하는가?
