# 세션 연속성 및 작업 지침

## 목적

이 문서는 새로운 작업 세션이 시작될 때 프로젝트의 맥락을 잃지 않고 이어가기 위한 운영 지침이다. 이 프로젝트는 문서 중심으로 방향을 잡고, 이후 Unity 기반 프로토타입을 단계적으로 만드는 방식으로 진행한다.

새 세션에서 AI 작업자는 먼저 `README.md`로 현재 상태와 다음 작업을 확인한다. 이 문서는 작업 절차와 협업 규칙이 필요할 때 읽는 운영 지침이다.

## 새 세션 시작 시 반드시 할 일

1. `README.md`를 읽어 현재 상태, 최신 방향, 다음 작업을 확인한다.
2. `docs/22-decision-log.md`를 읽어 확정 결정과 보류 사항을 확인한다.
3. 작업 성격에 따라 전담 문서를 추가로 읽는다.
   - 문서 역할 확인: `docs/05-design-document-roadmap.md`
   - 구현 작업: `docs/18-prototype-task-board.md`, `docs/15-prototype-scope-definition.md`, `docs/16-technical-design-brief.md`
   - 레퍼런스/치환 작업: `docs/06-eatventure-reference-analysis.md`, `docs/04-theme-translation-guide.md`
   - 장비/상자/조합 메타 작업: `docs/12-meta-progression-design.md`
4. 필요한 경우 `docs/work-history.md`를 읽어 최근 변경 이유를 확인한다.
5. 작업 후에는 의미 있는 변경 사항을 `docs/work-history.md`에 업데이트한다.

## 프로젝트 현재 방향

- 목표 게임: Eatventure의 직관적인 운영 성장 문법을 참고하되, 음식/레스토랑 운영이 아니라 전자/가전제품 생산, 조립, 판매, 자동화로 재해석한 Steam PC용 게임
- 핵심 판타지: 작은 작업대에서 시작해 직원, 조립 라인, 컨베이어, 자동화 설비를 확장하며 스마트 팩토리로 성장
- 현재 단계: Unity 프로젝트와 마일스톤 2 자동 작업 루프 골격이 완료되었고, 마일스톤 3 아이템 운반과 스테이션 처리를 준비 중
- 중요한 원칙: Eatventure를 그대로 복제하지 않고, 작동 원리만 구조적으로 추출해 새로운 테마와 시스템으로 변형한다.
- 조작 원칙: 플레이어는 WASD로 캐릭터를 직접 조종하지 않는다. 작업자와 스테이션이 자동으로 일하고, 플레이어는 클릭/UI로 투자, 해금, 업그레이드, 우선순위를 조정한다.
- 카메라 원칙: 위에서 수직으로 내려다보는 고정 탑다운 뷰를 기준으로 한다.
- 메타 원칙: 스테이지별 첫 작업자는 플레이어 고유 작업자이며, 장기적으로 보상 상자, 장비 장착, 장비 조합을 구현한다.

## 추천 기술 스택

현재 추천 기술 스택은 다음과 같다.

- 엔진: Unity
- 권장 버전: Unity 6 LTS 계열
- 언어: C#
- 렌더링: URP
- 게임 표현: 위에서 수직으로 내려다보는 3D 탑다운 고정 뷰
- 입력: Unity Input System
- 초기 프로토타입 입력: 스테이션 클릭과 UI 버튼 중심
- UI: 초기에는 uGUI 또는 Unity UI Toolkit 중 프로젝트 편의에 따라 선택
- 데이터: ScriptableObject 기반으로 시작, 필요 시 JSON/CSV 확장
- 저장: 초기에는 로컬 JSON 세이브, 출시 준비 단계에서 Steam Cloud 검토
- Steam 연동: 초기 프로토타입 단계에서는 제외, 출시 준비 단계에서 Steamworks.NET 검토
- 버전 관리: Git, 에셋이 늘어나면 Git LFS 사용

## 기술 스택 판단 근거

Unity를 추천하는 이유:

- 캐주얼 운영/타이쿤/자동화 게임의 프로토타입을 빠르게 만들기 좋다.
- Steam PC 출시 사례와 자료가 많다.
- 2D, 2.5D, 간단한 3D를 모두 수용할 수 있다.
- C# 기반이라 구조화된 시스템 구현과 유지보수에 유리하다.
- Steamworks 연동 플러그인과 커뮤니티 레퍼런스가 많다.
- 이후 필요하면 모바일 확장 가능성도 남길 수 있다.

Unreal은 현재 프로젝트 범위에는 과할 가능성이 높다. Godot도 가능하지만, Steam 출시 자료, 에셋 생태계, Unity 경험자 풀, 상용 파이프라인 측면에서 Unity가 더 안전한 기본 선택이다.

## 사용자 특성 및 협업 방식

프로젝트 오너는 Unity를 처음 다루는 초보자이며, AI의 도움을 받아 문서, 코드, 구조, 구현을 함께 진행하는 바이브 코딩 방식에 가깝다.

따라서 AI 작업자는 다음 방식으로 협업해야 한다.

- Unity 경험이 없는 사람도 따라올 수 있게 단계별로 설명한다.
- 전문 용어를 사용할 때는 필요한 만큼 짧게 뜻을 설명한다.
- 한 번에 너무 많은 Unity 개념을 요구하지 않는다.
- 사용자가 직접 해야 하는 에디터 조작은 명확한 순서로 안내한다.
- 가능하면 반복 작업은 코드, 에디터 스크립트, 프리팹 구조로 자동화하는 방향을 고려한다.
- 사용자가 화면 캡처, 콘솔 에러, 동작 설명을 제공하면 그 정보를 기준으로 원인을 좁힌다.
- 사용자가 모르는 것을 전제로 하되, 과도하게 낮춰 설명하지 않는다.

## AI 작업자가 할 수 있는 일

- 기획 문서 작성 및 업데이트
- 프로토타입 범위 정의
- Unity 프로젝트 구조 제안
- C# 스크립트 작성
- 데이터 구조 설계
- 씬 구성 지시서 작성
- Unity 콘솔 에러 분석
- Git 상태 확인 및 작업 범위 정리
- Steam 출시 준비 체크리스트 작성
- 작업 후 히스토리 기록

## AI 작업자가 혼자 확정하면 안 되는 일

다음 항목은 프로젝트 방향에 큰 영향을 줄 수 있으므로, 사용자의 의도가 불명확하면 반드시 되물어야 한다.

- 엔진 변경
- 목표 플랫폼 변경
- 게임 장르 또는 핵심 루프 변경
- Eatventure 참고 강도와 차별화 방향의 큰 변경
- 유료/무료, BM, DLC, 얼리 액세스 등 상품 전략 확정
- 아트 스타일의 큰 방향 확정
- 프로토타입 범위의 대폭 확대 또는 축소
- 저장 구조, 네트워크, 온라인 기능처럼 장기 구조에 영향을 주는 결정
- 기존 문서의 핵심 방향을 뒤집는 수정
- 사용자가 명시하지 않은 파일 삭제 또는 대규모 정리

## 이해가 불확실할 때의 규칙

AI 작업자는 사용자의 요청이 모호하거나, 해석에 따라 결과가 크게 달라질 수 있거나, 장기 방향에 영향을 줄 수 있다고 판단하면 혼자 추측해서 진행하지 않는다.

이 경우 반드시 다음 형식으로 되묻는다.

- 내가 이해한 요청
- 헷갈리는 지점
- 선택 가능한 방향
- 추천 방향이 있다면 그 이유

단, 작은 문서 정리, 오탈자 수정, 명확한 후속 작업처럼 리스크가 낮은 작업은 합리적으로 판단해 진행한다.

## 작업 후 기록 규칙

모든 의미 있는 작업 후에는 `docs/work-history.md`를 업데이트한다.

기록에는 다음을 남긴다.

- 날짜
- 작업 요약
- 생성/수정한 파일
- 결정된 사항
- 아직 미정인 사항
- 다음 작업 후보
- 사용자의 중요한 선호 또는 지시

기록은 길게 쓰기보다 다음 세션에서 빠르게 이어받을 수 있을 정도로 명확하게 쓴다.

## 프로토타입으로 넘어가기 전 권장 순서

1. `docs/15-prototype-scope-definition.md` 작성
2. Unity 버전 확정
3. Unity 프로젝트 생성
4. 첫 프로토타입의 최소 루프 구현
   - 부품대, 조립대, 판매대 배치
   - 작업자 자동 이동
   - 작업자 부품 픽업
   - 조립대 투입
   - 제품 생성
   - 작업자 제품 회수와 판매/출고
   - 돈 획득
   - 스테이션 레벨업 또는 업그레이드 1개
   - 추가 직원 또는 자동화 설비 구매
5. 프로토타입 플레이 후 재미, 이해도, 운영 개입 피로도 기록
