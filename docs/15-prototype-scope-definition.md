# 첫 프로토타입 범위 정의서

## 문서 목적

이 문서는 Unity로 만들 첫 프로토타입의 범위, 제외 항목, 검증 질문, 성공 기준을 정의한다. 목표는 최종 게임의 축소판을 만드는 것이 아니라, 이 게임의 핵심 재미가 성립하는지 가장 작은 단위로 확인하는 것이다.

현재 프로토타입은 조사 완료 후 확정된 설계가 아니라, 기존 초기 기획 문서를 바탕으로 한 가설 검증용이다.

## 프로토타입 한 줄 목표

플레이어가 부품을 가져와 제품을 조립하고 판매해 돈을 벌며, 업그레이드와 직원 자동화를 통해 반복 작업이 줄어드는 핵심 운영 루프를 Unity에서 직접 체감한다.

## 검증하려는 핵심 가설

1. 전자/가전 생산 테마는 음식 조리 테마처럼 직관적으로 이해될 수 있다.
2. 부품 픽업, 조립, 판매, 업그레이드의 짧은 루프는 반복 플레이의 기본 재미를 만들 수 있다.
3. 플레이어 직접 작업에서 직원 자동 작업으로 넘어가는 순간은 충분한 성장 체감을 만든다.
4. 단순한 3D 탑다운/아이소메트릭 표현만으로도 생산 라인의 변화와 자동화 쾌감을 전달할 수 있다.
5. Steam PC용 게임으로 확장하려면 어떤 조작, UI, 자동화 깊이가 필요한지 초기 판단을 할 수 있다.

## 프로토타입 기준 플레이 경험

플레이어는 작은 전자제품 작업장에 있다. 작업장에는 부품 보관대, 조립대, 판매대, 업그레이드 버튼, 직원 구매 버튼이 있다.

플레이어는 부품 보관대에서 부품을 하나 들고 조립대로 이동한다. 조립대에 부품을 넣으면 일정 시간 후 간단한 제품이 완성된다. 완성품을 판매대에 놓으면 돈을 얻는다. 돈을 모아 조립 속도 또는 판매 가격을 업그레이드하고, 일정 금액이 모이면 직원 한 명을 고용한다. 직원은 부품 운반 또는 판매 작업 중 하나를 자동으로 수행한다.

이 흐름이 5~10분 안에 반복되고, 플레이어는 업그레이드 전후의 차이를 체감해야 한다.

## 포함 범위

### 1. 기본 씬

포함:

- 하나의 테스트 작업장 씬
- 바닥, 벽 또는 경계
- 카메라
- 조명
- 플레이어 캐릭터
- 부품 보관대
- 조립대
- 판매대
- 업그레이드 지점 또는 UI
- 직원 1명

제외:

- 여러 공장
- 여러 층 또는 방
- 완성된 인테리어
- 최종 아트 리소스

### 2. 플레이어 이동

포함:

- 키보드 이동
- 마우스 또는 키보드 기반 상호작용
- 간단한 탑다운 카메라
- 오브젝트 근처에서 상호작용 가능

기본 가정:

- 초기에는 WASD 이동을 사용한다.
- 마우스 클릭 이동은 후순위로 둔다.
- 조작감 검증이 목적이므로 캐릭터 애니메이션은 최소화한다.

### 3. 아이템 운반

포함:

- 플레이어가 부품 1개를 들 수 있다.
- 플레이어가 완성품 1개를 들 수 있다.
- 현재 들고 있는 물건이 시각적으로 표시된다.
- 들고 있는 상태에 따라 상호작용 대상이 달라진다.

제외:

- 여러 개 동시 운반
- 인벤토리
- 아이템 스택
- 복잡한 물류 우선순위

### 4. 생산 루프

포함:

- 부품 보관대에서 부품 생성 또는 무한 제공
- 조립대에 부품 투입
- 조립 시간 대기
- 완성품 생성
- 완성품을 판매대로 이동
- 판매 시 돈 획득

초기 제품:

- 가칭: `Basic Gadget`
- 의미: 스마트폰, 이어폰, 보조배터리 중 하나로 확정하기 전 사용하는 임시 전자제품
- 프로토타입에서는 외형보다 루프 검증을 우선한다.

제외:

- 제품군 다중화
- 부품 종류 다중화
- 검사 공정
- 포장 공정
- 결함률
- 품질 등급

### 5. 경제와 업그레이드

포함:

- 돈 표시
- 제품 판매가
- 조립대 업그레이드 1종
- 판매가 업그레이드 1종 또는 조립 속도 업그레이드 1종
- 직원 구매 비용
- 업그레이드 구매 후 즉시 효과 반영

초기 수치 가설:

| 항목 | 초기값 |
| --- | --- |
| 제품 판매가 | 5 |
| 조립 시간 | 3초 |
| 조립 속도 업그레이드 비용 | 20 |
| 조립 속도 업그레이드 효과 | 조립 시간 20% 감소 |
| 판매가 업그레이드 비용 | 30 |
| 판매가 업그레이드 효과 | 판매가 +5 |
| 직원 구매 비용 | 50 |

수치는 재미 검증용 임시값이다. 최종 밸런스로 보지 않는다.

### 6. 직원 자동화

포함:

- 직원 1명 구매 가능
- 직원은 정해진 작업을 자동 반복한다.
- 첫 버전 직원 역할은 하나만 선택한다.

권장 직원 역할:

- 운반 직원: 부품 보관대에서 부품을 가져와 조립대에 넣는다.

선택 이유:

- 플레이어가 하던 반복 작업 일부가 사라지는 자동화 체감을 가장 빠르게 보여준다.
- 조립대가 계속 돌아가는 느낌을 만들기 쉽다.
- 판매까지 자동화하면 플레이어가 너무 빨리 할 일이 없어질 수 있다.

제외:

- 직원 여러 명
- 직원 레벨업
- 직원 역할 변경 UI
- 직원 경로 최적화
- 매니저 시스템

### 7. UI/HUD

포함:

- 현재 돈
- 현재 들고 있는 아이템
- 조립대 상태
- 업그레이드 비용
- 직원 구매 비용
- 간단한 안내 텍스트

제외:

- 최종 HUD 디자인
- 복잡한 메뉴
- 옵션 화면
- 세이브 슬롯 UI
- 튜토리얼 시스템

### 8. 저장

첫 프로토타입에서는 저장 기능을 제외한다.

이유:

- 핵심 재미 검증에는 저장이 필요하지 않다.
- 저장 구조를 성급히 만들면 이후 데이터 구조 변경 비용이 생긴다.
- 프로토타입 1차 목표는 5~10분 플레이 루프 확인이다.

단, 두 번째 프로토타입부터는 로컬 JSON 저장을 검토한다.

이번 프로토타입에서 명시적으로 하지 않는 것:

- Steam Cloud
- 업적
- 데모 빌드
- 여러 제품군

- 연구 시스템
- 장비/스킨/코스메틱
- 이벤트
- 메타 진행
- 사운드 완성
- 최종 아트
- 튜토리얼 완성
- 저장/불러오기
- 모바일 대응
- 컨트롤러 대응
- 정교한 경제 밸런스

## 프로토타입 성공 기준

### 최소 성공

다음이 가능하면 최소 성공으로 본다.

- Unity에서 씬이 실행된다.
- 플레이어가 이동할 수 있다.
- 부품을 가져와 조립대에 넣을 수 있다.
- 조립 시간이 지난 뒤 제품이 나온다.
- 제품을 판매대에 놓으면 돈이 증가한다.
- 돈으로 업그레이드를 구매할 수 있다.
- 직원 1명이 자동으로 반복 작업을 수행한다.

### 의미 있는 성공

다음이 느껴지면 의미 있는 성공으로 본다.

- 플레이어가 1분 안에 무엇을 해야 하는지 이해한다.
- 업그레이드 후 작업 속도 또는 수익 차이가 체감된다.
- 직원 구매 후 반복 작업이 줄어드는 느낌이 있다.
- 화면에서 생산 흐름이 읽힌다.
- 5분 이상 반복해도 다음 업그레이드를 사고 싶다는 동기가 있다.

### 실패로 볼 신호

다음이 나타나면 설계 재검토가 필요하다.

- 부품, 조립, 판매 관계가 직관적으로 이해되지 않는다.
- 이동이 지루하고 조작 피로가 먼저 느껴진다.
- 자동화 전까지 너무 단조롭다.
- 직원 구매 후 플레이어가 할 일이 사라진다.
- 업그레이드 효과가 숫자로만 느껴지고 행동 변화가 없다.
- 전자/가전 테마가 음식 게임의 스킨처럼 보인다.

## 테스트 플레이 체크리스트

테스트 후 다음 질문에 답한다.

- 첫 30초 안에 목표가 보였는가?
- 첫 판매까지 시간이 너무 길거나 짧지 않았는가?
- 조립 시간은 기다릴 만했는가?
- 업그레이드를 사고 싶다는 생각이 들었는가?
- 직원 구매 전후 차이가 명확했는가?
- 직접 조작과 자동화의 비율이 적절했는가?
- 화면에서 어떤 오브젝트가 어떤 역할인지 알 수 있었는가?
- 전자제품을 만든다는 느낌이 있었는가?
- 10분 뒤에도 확장하고 싶은 마음이 있었는가?
- 다음에 추가해야 할 시스템이 자연스럽게 떠올랐는가?

## 첫 구현 순서 제안

1. Unity 프로젝트 생성
2. 기본 씬 생성
3. 플레이어 이동 구현
4. 상호작용 범위와 상호작용 입력 구현
5. 부품 보관대 구현
6. 아이템 들기/내려놓기 구현
7. 조립대 구현
8. 판매대와 돈 획득 구현
9. HUD 구현
10. 업그레이드 1종 구현
11. 직원 1명 자동 작업 구현
12. 테스트 플레이 후 수치 조정

## Unity 초보자 기준 에디터 작업 범위

사용자가 직접 해야 할 가능성이 높은 작업:

- Unity Hub에서 프로젝트 열기
- Play 버튼으로 실행
- Console 창의 에러 확인
- Scene/Game 뷰 확인
- 오브젝트 위치가 이상할 때 화면 캡처 제공

AI가 우선 담당할 작업:

- 폴더 구조 제안
- C# 스크립트 작성
- 오브젝트 구성 방식 설명
- 필요한 경우 에디터에서 붙일 컴포넌트 목록 제공
- 오류 메시지 분석
- 다음 수정안 작성

## 미정 사항

다음은 구현 전 또는 구현 중 사용자 확인이 필요할 수 있다.

- Unity 정확한 버전
- 프로젝트 이름
- 초기 제품을 스마트폰, 이어폰, 보조배터리 중 무엇으로 표현할지
- UI를 uGUI로 시작할지 UI Toolkit으로 시작할지
- 플레이어 이동을 WASD로 고정할지 마우스 클릭 이동도 함께 고려할지
- 3D 탑다운과 아이소메트릭 중 카메라 기준을 어디에 둘지
